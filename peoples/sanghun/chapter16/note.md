### ch16 코드 정리 구분
동작 변경을 위해 코드를 읽다 보면 구조 변경을 하고 싶은 부분들이 보인다. 코드 이해를 쉽게 하기 위해서나 동작 변경의 복잡성을 제거하기 위해서 따위의 이유로 구조 변경은 어느 정도 포함됐었다. 하나의 PR에.

구조 변경과 동작 변경을 구분한 적은 없었다. PR을 작게 작성하길 원하면서도, 변경 내용이 많더라도 task 하나에 PR 하나면 충분하지 않나 라며 편하게 생각했었다. 더 나눌 기준을 생각해보진 않았었다. 책에서는 변경의 순서에 대한 내용이 있다. 순서가 있다는 것은, 이전의 변경은 다음 변경을 위한 목적이 분명한 것을 의미한다고 이해했다. 

보통 PR 검토는 진행 중인 작업을 끝내거나 한 부분을 정리했을 때 진행했다. 브랜치를 변경하고 새로운 로직을 읽고 이해하는 일을 작업 중간에 수시로 전환하긴 힘든 부분이 있었기 때문이다. 그러다 보면 검토 시점이 밀리고 PR이 쌓이거나 하는 일도 있었다. 검토 품질도 일정하지 않았다.

구조 변경과 동작 변경을 구분하고 분명한 초점을 가지고 있는 PR을 발행한다면 빠르고 품질 높은 검토와 통합 비용을 절감을 얻을 수 있을 것 같다. 물론 PR 작을 수록 무조건적으로 좋다는 뜻은 아니다. 수와 양의 적정 선은 팀의 합의에 의해야 한다.
