# Chapter 18: 코드 정리의 일괄 처리량
코드 정리를 한번에 많이 해서 반영하려 하면, 그에 따라오는 영향으로 인한 비용도 커진다. 예컨데, 다른 사람이 작업하고 있는 부분을 건드리게 될 수도 있고, 다른 코드의 동작이 바뀔수도 있고, 변경사항을 반영하기 전에 하는 테스트에서 시간이 더 오래 걸릴수도 있다. 그러면 잘게 쪼개서 코드 정리를 반영하려 하면? 각 반영 건 마다 검토를 해야 하는데 그 숫자가 또 너무 많으면 검토하는 사람 입장에서도 부담이 되고 이에 소요되는 시간도 늘어날 수 있다. 결론적으로는 검토 비용을 줄이는 방법을 찾으하는 내용이였다. 예컨데 PR 을 통해 검토를 한다면, PR 하나 검토하는 데 드는 시간, 부담, 분량 등을 줄여야 한다는 것이다. 각 프로젝트 마다 어떻게 하느냐에 따라 다르겠지만, PR 작성에 대한 가이드나 템플릿 등이 있으면 아마 도움이 될지도 모르겠다. PR 하나에는 하나의 토픽만, 대략 최대 몇줄 정도 까지만 허용 할 것인지 가이드를 넣을수도 있고. 테스트 자동화를 해서 검토해야 할 부분을 줄이는 방법도 있겠다(물론 )
