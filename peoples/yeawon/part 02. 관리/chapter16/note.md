## 16장. 코드 정리 구분

- PR / 코드 검토 모델을 사용한다고 가정하자. 코드 정리는 어디서 해야 할까?
- 코드 정리는 <u>동작 변경 코드와 분리</u>하여 **별도의 PR로 만들고, 가급적 PR당 몇 개의 코드 정리만 넣는다**

- 코드 정리를 배우는 사람들은 다음과 같이 예상되는 단계들을 거치게 된다
    1. **변경을 구분하지 않은 상태에서 다수의 변경을 반영하기 시작**
        - ex) if문을 변경하는 도중에 이름이 잘못되었다는 것을 깨닫고 이름을 변경한 후 다시 if문으로 돌아가는 경우

    2. **동작 변경과 구조 변경**
        - **동작 변경**
            - 프로그램 동작 변경. 프로그램을 실행하면서 찾아낸 것
            - 저자: *"프로그램을 구성하는 코드를 보지 않아도 결과가 드러나는 변경을 말한다. 외부의 관찰자에게 보인다는 말"*
        - **구조 변경**
            - 프로그램 구조 변경. 코드를 자세히 살펴봐야만 관찰할 수 있음
            - 저자:*"파일의 형식을 바꾸는 일은 또 다른 프로그래머만 관찰할 수 있기 때문에 동작 변경이 아니다. 저는 이런 경우를 구조 변경이라고 부릅니다."*
            > ❓ 동작 변경은 기능 추가, 수정 등을 말하는 거고 구조 변경은 이 책에서 설명하는 코드 정리법들을 말하는 걸까?
        - 이 시점에서는 **아직 계획도**, 바뀌는 동작 변경과 구조 변경 사이에 **어떤 흐름도 없음.** <u>서로 다른 두 가지가 함께 작용하고 있다는 것을 인식했을 뿐</u>

    3. **순서를 부여한 동작 변경과 구조 변경**
        - 위의 과정이 조금 지나면, **공통 흐름을 알아채기 시작한다.** 
        - <u>비슷한 코드끼리 정리</u>하면 → <u>도우미가 드러나고</u>, → <u>설명하는 도우미를 만들면</u> → **이어지는 동작 변경이 훨씬 쉬워진다** ⇒ 이렇게 되면, 어떤 순서로 전개될지 보이고, 몇 수 앞을 짐작할 수 있다
        - **그러나 아직 큰 PR 한 개에 섞여 있을 뿐임을 주의하라.** 아직 반복과정 1단계에 있다
        - <u>쉽게 변경하든가</u> or <u>변경하기 쉽게 만들든가</u>, 둘 중 하나를 목표로 삼을 것

    4. **별도의 PR에 포함된 동작 변경과 구조 변경**
        - 그러니, **우리는 변경사항을 나누어 별도의 PR로 만들어야 한다.**       
            - 순서가 있는 일련의 코드 정리는 하나의 정리만 있더라도 PR 한 개로 만든다.
            - 동작 변경 역시 별도의 PR로 만든다
        - **코드 정리와 동작 변경 사이를 번갈아 가면서 전환할 때마다 새 PR을 연다**
            > ❓ 이게 현실적으로 가능할까? 동작 변경을 한 PR에서 마무리하고, 그 이후에 코드 정리 PR을 따로 진행하는 방식으로 해야 하지 않을까..?

- <u>이렇게 PR을 나눌지</u> vs <u>한 개의 PR로 처리할지</u>는 **장단점**이 있다
    - **크고 포괄적인 PR**: 전체 그림을 보여 주지만, 검토하는 입장에서는 유용한 피드백을 제공하기에 너무 큰 덩어리일 수 있다
    - **아주 작은 PR**: 소소한 피드백을 유도할 수 있지만, 무시될 염려도 있다

- 작은 PR은 코드를 신속하게 검토할 수 있다. 초점이 분명할 수록 더 빠른 검토 권장
- 코드 정리에 익숙해지고, 작은 단위로 작업하고, 절대적으로 안전하게 작업하는 데 익숙해지면, 코드 정리 PR은 검토하지 않는 시도를 해봐라 → 작은 PR을 만들도록 장려하는 효과가 생길 것!
