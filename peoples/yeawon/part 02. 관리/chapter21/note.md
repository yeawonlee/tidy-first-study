## 21장. 코드 정리 시점

- 코드 정리를 먼저 하고 나서 동작 변경을 할까? 아니면 동작 변경을 먼저 한 후, 코드 정리를 할까? 메모로 남기고 정리를 미룰까? 아예 코드 정리를 하지 말까?

### 1. 코드 정리를 아예 안 한다면
- 앞으로 다시는 코드를 변경하지 않을 때
- 설계를 개선하더라도 배울 것이 없을 때

### 2. 나중에 정리하기
지금 변경한 동작과 연결되지 않는 코드 정리를 나중에 하면 몇 가지 다른 방식으로 가치를 창출한다.
- **정리할 코드 분량이 많은데, 보상이 바로 보이지 않을 때 - 지저분함 보유세(tax of messiness)를 줄이기**
    - ex) 이전 API에서 새 API로 바뀌는 과정을 생각해 보자. 당장 영향 받는 API 호출 코드는 변경했지만, 나중에 변경을 보유해야 할 곳이 100개 더 있다. 변경을 모두 완료하여 이전 API를 제거할 수 있게 되기 전까지는, 새 API에 변경이 발생하면 이전 API에도 똑같이 반영해야 하는 불편함 존재
- **코드 정리에 대한 보상이 잠재적일 때**
- **학습 도구로 활용하는 것**
   - 코드 정리는 설계한 세부 결과를 깨달을 수 있는 좋은 방법이다. 코드 정리는 생각할 수 있는 설계를 비춰준다
- **작은 묶음으로 여러 번에 나눠서 코드 정리를 할 수 있을 때**
- 자신이 원할 때 하면 더 의욕적이고 기분이 좋다

### 3. 동작 변경 후에 코드 정리
- 동작은 변경해야 하는데 코드가 지저분할 경우, 어떻게 코드 정리해야 할지 당장은 모르겠으니 동작을 변경한다.이제 코드를 정리해야 할까?
- 상황에 따라 다르다. 방금 변경한 코드를 대상으로 다시 동작 변경을 하게 될까? 같은 영역을 다시 변경하게 된다면, 동작 변경 후 코드 정리하는 일은 상당한 의미가 있다.
- 나중에는 맥락을 잊어버리거나 또 다른 변경 건이 생겨 코드 정리에 방해를 받을 수 있기 때문에,나중에 코드를 정리할 때까지 기다리느라 비용이 너무 많이 증가할 가능성이 있다면, 지금 정리하는 것이 좋다
- 그런데 정리는 어느 정도나 해야 할까? 동작 변경에 한 시간이 걸렸다면 한 시간 정도 코드 정리에 투자하는 것은 합리적. 그러나 그 이상은 나중에 정리할 목록에 두자
- **다음의 경우라면 동작 변경 후 코드 정리하라**
    - **방금 고친 코드를 다시 변경할 예정일 때**
    - **지금 정리하는 것이 더 저렴할 때**
    - **코드 정리하는 데 드는 시간 ≒ 동작 변경에 드는 시간**
    - 코드 정리를 하지 않으면 일을 끝냈다는 느낌이 들지 않을 때

### 4. 코드 정리 후에 동작 변경
코드 정리를 선행해야 할까? → 상황에 따라 다르다
- **코드 정리를 했을 때, 코드 이해가 쉬워지거나 동작 변경이 쉬워지는 즉각적인 효과를 얻을 수 있을 때**
    - 지저분한 상태 그대로 코드를 변경한다면, 일이 얼마나 더 어려운가? 코드를 정리한다고 해서 더 쉬워지지 않는다면, 먼저 정리하지 마라
    - 코드 정리의 이점을 바로 얻을 수 있는가? 아직 동작 변경 준비가 되지 않았고 이해하고자 코드를 읽고 있을 때, 코드를 정리하면 더 빨리 이해할 수 있겠다는 판단이 서면 코드 정리를 먼저 하라
- **어떤 코드를 어떻게 정리해야 하는지 알고 있을 때**
- **코드 정리에 드는 비용을 보상 받을 수 있을 때**
    - 이 코드를 딱 한 번만 변경할 예정이라면 코드 정리는 제한하는 것이 좋다
- **코드 정리에 대한 확신이 있을 때** (추측할 때는 편견에서 벗어나야 한다)

---

> **내 경험 💬)** 
> - 나는 나중에 정리하기를 택한 적도 있고, 한번 정리할 때 관련된 코드를 최대한 리팩토링 하려고 했던 것 같다. 그래서 하다 안 되면 원복시킨 적도 있고, 코드 정리하는 데 시간이 더 들었던 적도 있던 것 같다.
> - 동작 변경 후에 코드를 정리한 경우와 코드 정리 후 동작 변경을 한 경우도 많았던 것 같다. 그러나 동작 변경과 코드 정리 PR을 잘 구분하지 않았던 것 같다
