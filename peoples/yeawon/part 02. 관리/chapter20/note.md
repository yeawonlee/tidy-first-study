## 20장. 얽힘 풀기

- 어떤 코드의 동작 변경을 하고 있다고 가정해 보자. 변경을 쉽게 할 코드 정리를 알고 있어 코드 정리를 한다. 새로운 테스트 케이스를 작성한다. 이제 동작을 변경해야 한다. 그랬더니 코드 정리할 곳이 더 생겼다
- 변경 대상인 동작을 모두 알게 되었고, 그 동작들을 쉽게 변경하려면 어떤 코드를 정리해야 하는지도 모두 알게 되었으나, **문제는 정리한 코드와 변경할 동작이 함께 얽혀버렸다**
- 3가지 선택
    1. 그대로 배포하기
    2. 코드 정리와 변경사항을 별도의 하나 이상의 PR로 나누거나 하나의 PR을 여러 번의 커밋으로 나누기
    3. **진행 중인 작업을 버리고, 코드 정리를 선행하는 순서로 다시 시작하기**. 이렇게 하면 작업은 더 많아지지만, 이어지는 커밋과의 일관성은 분명해진다
- 왜 3번을 선택해야 하는가? **우리가 작성하는 코드는 컴퓨터에 지시할 뿐 아니라, 컴퓨터에 지시하려는 우리의 의도를 사람들에게 설명해야 한다**
- **다시 구현하면서 새로운 것을 발견할 가능성이 높아진다. 그리고 동일한 동작 변경을 하면서도 더 많은 가치를 끌어낼 수도 있다**
    > **내 생각 💬)** 일리가 있는 얘기지만, 팀원들 모두가 동의할 때 진행할 수 있지 않을까? 그런데 한편으로는 코드 정리는 앞서 말했듯이 최대 1시간을 넘기지 않는 것을 지킨다면 납득할 수 있을 것 같다.
- **실타래를 풀어야 할 필요성을 더 일찍 깨달을수록 작업량은 적어질 것**
- 코드 정리와 동작 변경 사이의 선후 문제는 보통 시간이 지나면 해결된다
