## 18장. 코드 정리와 일괄 처리량

- 통합과 배포를 하기 전에 코드 정리는 어느 정도 크기가 적절할까?
- 몇 가지 고려할 사항
    - 코드 정리를 다음 동작 변경을 돕는 구조 변경으로 본다면, 다음 동작 변경을 지원하기 위해 얼마나 많은 구조 변경이 필요할까요? (기본적으로 코드 정리는 즉각적인 필요를 다뤄야 한다)
    - 코드 정리의 크기가 어느 정도일 때 쉽게 통합하고 배포할 수 있을까?

- `일괄 처리당 코드 정리 수(규모)` ↑ → `충돌, 상호작용, 추측` ↑
- 일괄 처리하는 코드 정리의 적절한 범위를 판단할 때, 고려할 만한 비용
    - 충돌:
        - 일괄 처리하는 코드 정리 작업 ↑ → 통합 과정에서 지연 시간 ↑ → 코드 정리 작업이 다른 사람의 진행 중인 작업과 충돌할 가능성 ↑
        - 코드 병합 과정에서 충돌이 발생할수록 → 병합하는 비용 또한 큰 폭으로 증가
    - 상호작용: 다수의 코드 정리를 한번에 처리하다가 우연히 동작 변경을 할 수도 있다. 마찬가지로 코드 정리 사이에 상호작용이 있으면 병합 비용은 급격히 증가
    - 추측: 앞서, 다음 동작 변경에 도움이 될 만큼만 코드를 정리하는 것이 좋다고 했다. 하지만, 한 번에 처리하는 코드 정리 多 → 자연스럽게 더 많은 코드를 정리하게 된다. 그로 인해 예상치 못한 추가 비용이 발생한다

- **이러한 요인으로 인해, 보통 한 번에 처리하는 코드 정리 개수를 줄인 후, 통합과 배치를 임한다**

- **많은 조직에서 하나의 변경 사항을 검토하고 배포하는 데 드는 고정 비용은 상당히 많다.** ∴ 프로그래머는 이러한 비용을 체감하기 때문에 <u>충돌, 상호작용, 추측으로 인한 비용이 증가됨에도</u>, **타협점을 찾으려고 한다**

- **코드 정리 비용을 줄이고자 한다면, <u>코드 정리 개수를 늘려서 동작 변경에 소용되는 비용을 줄여라.</u> 그러면, 검토 비용을 줄일 수 있다.**

⇒ **`일괄 처리 규모를 줄이면` ↓ → `검토 비용` ↓, `코드 정리 비용` ↓**

- **팀 내에서 검토 비용을 제대로 줄일 수 있는 방법을 찾아야 한다**
- 코드 정리 검토를 없앨 수준의 안전과 신뢰에 도달하려면, 적어도 몇 달이 걸림. 실천하고 실험하라. QA 팀(이 없지만..)과 함께 검토하라
