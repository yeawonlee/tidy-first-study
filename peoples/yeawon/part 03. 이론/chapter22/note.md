## 22장. 요소들을 유익하게 관계 맺는 일
**소프트웨어 설계 = 요소들을 유익하게 관계 맺는 일**

### 요소
물질의 구조는 다음과 같은 부분들로 이루어진다
- 생물: 세포소기관 → 기관 → 유기체
- 물질: 원자 → 분자 → 결정
- 프로그래밍 세계: 토큰 → 식(expression) → 문(statement) → 함수 → 객체/모듈 → 시스템
    - ex) `const total = price * count;`
        - **토큰** → `const`, `total`, `=`, `price`, `*`, `count`
        - **식(expression)** → `price * count`
        - **문(statement)** → `const total = price * count`
        - **함수** → 여러 문이 모여 함수
        - **모듈** → 여러 함수
        - **시스템** → 여러 모듈

요소에는 경계가 있다. 그래서 어디서 시작하고 끝나는지 알 수 있다.
```js
function calculateTotal(price, count) {
  return price * count;
}
```
> - 경계: 함수 시작 `{`, 함수 끝 `}`
>- 이 함수 안에 무엇이 있는지, 밖에 무엇이 있는지 알 수 있다.

또한 요소는 하위 요소를 포함한다. 계층 구조를 가진다
```
시스템
 └ 모듈
    └ 함수
       └ 문
          └ 식
             └ 토큰
```
프로그래머들은 보통 Composite 패턴과 같은 동질적인 계층 구조를 선호한다. 
```
컴포넌트
 └ 컴포넌트
    └ 컴포넌트
```
하지만, 앞에 물질의 예시처럼 자연의 계층 구조는 동질적이지 않다. 하위 요소는 이들을 포함하는 요소와 다르다.

### 관계 맺기
요소로 된 계층 구조가 생겼다. **이러한 요소들은 서로 관계를 지닌 존재들이다.** 하나의 함수가 다른 함수를 호출한다. 이때, 함수가 바로 그 요소를 이뤄서 '호출하고/호출받는' 관계를 맺는다.
소프트웨어 설계에서의 관계는 다음과 같은 것들이 있다.
- 호출
- 발행 (Publish)
- 대기 (Listen)
- 참조 (변수의 값을 가져오기)

### 유익하게
- 하나의 설계 작업은 작은 하위 요소로 만든 거대한 수프를 만드는 일과 같다.
- 설계할 때 시스템에는 필요하지만, 기계를 위한 명령어가 아닌, 일종의 중간 요소를 만들면, 그 중간 요소들이 서로에게 도움이 되기 시작한다. 예를 들어, 함수 A는 함수 B가 계산의 복잡한 부분을 덜어 가면 더 간단해지는 것처럼 말이다

### 요소들을 유익하게 관계 맺는 일
- **설계 = 설계를 구성하는 요소들과 그들의 관계, 그리고 그 관계에서 파생되는 이점**
- **설계자는 요소들을 유익하게 관계 맺는 일을 한다.** 설계자는 다음과 같은 일만 할 수 있다.
    - 요소를 만들고 삭제한다
    - 관계를 만들고 삭제한다
    - 관계의 이점을 높인다

#### 예시) 한 객체가 하나의 함수에서 다른 객체를 두 번 호출하는 예시

```
caller()
    return box.width() * box.height()
```
> 호출 함수 `caller()`는 `box` 객체의 두 개의 함수를 호출하는 관계를 가진다

식을 box 객체 안으로 옮겨보자
```
caller()
    return box.area()

Box>>area()
    return width() * height()
```
> `Box.area()`라는 새로운 요소를 만들고 `caller()` 함수와 `box` 객체 사이의 관계를 조정했다. 이제 두 요소는 하나의 함수 호출로 연결된다. `caller()` 함수가 단순해진(유익해진) 대가로 `Box`가 하나의 더 커진 함수를 갖게 된다.

#### 시스템의 구조
- 요소 계층 구조
- 요소 사이의 관계
- 이러한 관계가 만들어내는 이점
